# Tarefa — Implementação de `jedi_library.db`

## Contexto

ADR conceitual: `jedi-library/docs/81-referencia/decisoes/20260612-jedi-db-criacao.md`.

ADR de implementação: `docs/81-referencia/decisoes/20260612-engine-migrations-sql-puro-tabela-schema.md`.

**Dependência:** `jedi_library.assets` (deve estar implementada antes).

## Entregáveis

### `jedi_library/db.py`

#### `open_connection(db_path: Path, *, foreign_keys: bool = True) -> sqlite3.Connection`

- `isolation_level = None` (autocommit desligado).
- `detect_types = PARSE_DECLTYPES | PARSE_COLNAMES`.
- PRAGMAs iniciais: `foreign_keys = ON` (se `foreign_keys=True`), `journal_mode = WAL`, `synchronous = NORMAL`.

#### `apply_migrations(conn, package, migrations_subdir="migrations") -> None`

1. Cria `schema_migrations(versao TEXT PRIMARY KEY, hash TEXT NOT NULL, aplicada_em TEXT NOT NULL)` se não existir.
2. Lista arquivos `V*.sql` via `pkg_files(package, migrations_subdir)`, ordena lexicograficamente.
3. Para cada arquivo não presente em `schema_migrations`:
   - `PRAGMA foreign_keys = OFF`
   - `SAVEPOINT migration_<versao>`
   - Executa statements do arquivo (splitar em `;`)
   - `RELEASE SAVEPOINT migration_<versao>`
   - `PRAGMA foreign_keys = ON`
   - `INSERT INTO schema_migrations(versao, hash, aplicada_em) VALUES (?, ?, ?)`
4. Em qualquer exceção: `ROLLBACK TO SAVEPOINT migration_<versao>`, re-raise com contexto.

### Testes (`test/python/test_db.py`)

- `open_connection` retorna conexão com FK ativado.
- `apply_migrations` cria `schema_migrations` + aplica migrations.
- Segunda chamada de `apply_migrations` não re-aplica nada (idempotência).
- Migration com erro: transação rolled back, linha não inserida em `schema_migrations`.
- Migration já aplicada mas com hash diferente: warning logged (não falha).

### Testes end-to-end (db em memória)

Usar `db_path = ":memory:"` ou `tmp_path` do pytest.
